\chapter{Testing and Evaluation}

This chapter presents the testing strategies and evaluation procedures used to verify the correctness, reliability, and performance of the Lets Go Ride Sharing App ride-sharing and recommendation system. Testing ensures that all functional and non-functional requirements defined in earlier chapters are satisfied. The chapter covers unit testing, integration testing, system testing, performance testing, security testing, model evaluation (for the ML recommendation component), and user acceptance testing.

% -------------------------------------------------
% 6.1 Testing Strategy
% -------------------------------------------------

\section{Testing Strategy}

The testing approach for the Lets Go Ride Sharing App system follows a multi-level strategy to ensure comprehensive validation at each stage of development. Both manual and automated testing techniques were employed.

\begin{itemize}
    \item \textbf{Unit Testing:} Individual modules and functions were tested in isolation using Python's \texttt{pytest} framework for backend services and Flutter's built-in test framework for frontend components. Mock objects were used to simulate external dependencies (e.g., database, APIs).
    \item \textbf{Integration Testing:} Interactions between integrated modules (e.g., backend APIs with database, frontend with backend) were tested using Postman for API endpoints and manual end-to-end verification of the Admin panel workflows. Database connectivity and transaction integrity were verified.
    \item \textbf{System Testing:} Complete end-to-end workflows covering all use cases (UC-1 to UC-25) were executed manually and through automated scripts. Edge cases, error handling, and user journeys were validated.
    \item \textbf{Performance Testing:} Load testing was performed using Apache JMeter to simulate multiple concurrent users and measure response times, throughput, and resource utilization under stress.
    \item \textbf{Security Testing:} Authentication mechanisms, role-based access control, input validation, and data encryption were tested using tools like OWASP ZAP and manual penetration tests.
    \item \textbf{ML Model Evaluation:} The content-based recommendation algorithm was evaluated offline using historical ride data, with metrics such as precision, recall, and F1-score.
    \item \textbf{User Acceptance Testing (UAT):} A group of potential users (students and staff) were invited to test the application and provide feedback, which was used to refine the UI and fix usability issues.
\end{itemize}

All testing activities were documented with test cases, expected results, and actual outcomes. A defect tracking system (GitHub Issues) was used to manage and resolve bugs.

\section{Integration Testing}

Integration testing validated the interactions between different modules and external services. Key integration points tested include:

\begin{itemize}

        \begin{itemize}
            \item Frontend Map Directions (OSRM/ORS): Road polyline and route directions retrieval for map visualization (with OSRM fallback when ORS is not configured).

\section{Security Testing}

Security testing was conducted using automated tools to assess both third-party dependency risk and source-code security patterns in the Django backend. A Django configuration sanity check was performed using \texttt{python manage.py check}, which reported no issues. Dependency vulnerability scanning was performed using \texttt{pip-audit} against \texttt{requirements.txt}. After upgrading vulnerable packages and re-testing, \texttt{pip-audit} reported ``No known vulnerabilities found''. Static application security testing (SAST) was performed using Bandit while excluding the virtual environment and test directories to reduce third-party noise. The Bandit scan covered 32,180 lines of code and reported 250 low-severity and 19 medium-severity findings, with no high-severity issues. Most low-severity findings were related to broad exception handling patterns (\texttt{try/except pass}) and were treated as robustness and maintainability concerns, while medium-severity findings highlighted areas for improvement such as reviewing network exposure configurations and security-related randomness usage in supporting scripts.

\section{Machine Learning Model Evaluation}

The content-based recommendation algorithm uses a custom weighted ranking score. Evaluation was performed using synthetic datasets generated to match Pakistan's urban topology.

\subsection{Ranking Logic Factors}
\begin{itemize}
    \item \textbf{Route Similarity (W=0.46):} Jaccard similarity of stop names and route pairs.
    \item \textbf{Soonness (W=0.12):} Exponential decay based on departure time relative to query time.
    \item \textbf{Driver Rating (W=0.18):} Impact of history and aggregate driver stars.
    \item \textbf{Gender Preference (W=Penalty):} Hard mismatch penalty for mixed-gender rides against user preference.
\end{itemize}
